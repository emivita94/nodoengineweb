Integración con ERP
Guía técnica completa — todo lo que el desarrollador necesita para integrarse
🔑
Credenciales de conexión
Estos son los únicos dos datos que el desarrollador necesita para empezar
URL BASE DE LA API
—
Todos los endpoints se construyen sobre esta URL. Por ejemplo, para emitir una factura: URL_BASE + /documentos/erp
API KEY — HEADER DE AUTENTICACIÓN
—
Esta clave identifica al emisor (empresa). Se envía en cada request como header HTTP: X-API-Key: sk_live_XXXX.
Sin este header, la API devuelve 401 Unauthorized. La clave es secreta — no exponerla en el frontend ni en logs.
⚠️ Ambiente de pruebas vs producción
Las keys que empiezan con sk_test_ operan contra el ambiente de pruebas de SIFEN (no envían documentos reales a la SET).
Las keys sk_live_ operan en producción — cada documento enviado queda registrado oficialmente en la SET.
Usá sk_test_ para todas las pruebas de integración antes de pasar a producción.

Integración en 5 pasos
Esta guía te lleva desde cero hasta emitir tu primera factura real en menos de una hora.
1
Configurar las credenciales en tu sistema
Guardá la URL base y la API Key que aparecen arriba en la configuración de tu ERP o sistema. No hardcodees estos valores en el código fuente — usá variables de entorno o un archivo de configuración.
# Variables de entorno recomendadas
NODO_API_URL=https://tu-api.railway.app/api/v1
NODO_API_KEY=sk_live_XXXX
2
Hacer el primer request de prueba
Antes de integrar en el código, verificá que la conexión funciona con este comando cURL desde tu terminal. Si devuelve 200 OK con el JSON de salud, la API está disponible.
Terminal / cURL
curl https://tu-api.railway.app/api/v1/../health

# Respuesta esperada:
{ "ok": true, "version": "0.1.0", "ambiente": "prod" }
3
Emitir un documento de prueba (cliente innominado)
El cliente innominado (tipo: 4) no requiere documento — es el más simple para probar. Usá una API Key sk_test_ para que no quede registrado en la SET real.
POST /api/v1/documentos/erp 
// Headers obligatorios en TODOS los requests:
Content-Type: application/json
X-API-Key: sk_test_XXXX

// Body mínimo — factura a innominado:
{
  "tipoDocumento": 1,       // 1 = Factura electrónica
  "cliente": {
    "tipo": 4              // 4 = Sin RUC/CI (innominado)
  },
  "items": [
    {
      "descripcion": "Producto de prueba",
      "cantidad": 1,
      "precioUnitario": 100000,
      "precioTotal": 100000,
      "tasaIVA": 10         // 0, 5 ó 10
    }
  ]
}
4
Interpretar la respuesta y guardar el CDC
Si el documento fue aprobado, la respuesta tiene estado: "aprobado" y un cdc de 44 dígitos. Guardá el CDC en tu base de datos — lo vas a necesitar para descargar el KUDE y consultar el estado.
201 Created Documento aprobado 
{
  "ok": true,
  "data": {
    "cdc":           "01800123456001002000001...", // ← GUARDAR EN BD
    "numero":        "001-002-0000001",           // Nro. visible en el KUDE
    "estado":        "aprobado",
    "sifen_codigo":  "0260",                      // 0260 = SET aprobó
    "sifen_mensaje": "Documento aprobado"
  }
}
5
Descargar el KUDE (PDF) y entregarlo al cliente
Con el CDC obtenido en el paso anterior, descargá el PDF. El endpoint devuelve el binario del PDF directamente. También podés hacer que NODO lo envíe automáticamente por email al cliente si configuraste SMTP.
GET /api/v1/documentos/{cdc}/kude?formato=a4 
// Devuelve el PDF binario (Content-Type: application/pdf)
// formato: a4 (defecto) | carta | ticket
// Requiere el header X-API-Key igual que el resto
✅ ¿Todo funciona? Ya podés pasar a las tabs siguientes para ver todos los endpoints en detalle, ejemplos en distintos lenguajes y el manejo de errores.
Emitir una factura electrónica
Este es el endpoint principal. Recibe los datos de la transacción, genera el XML según la norma SIFEN, lo firma con el certificado digital del emisor y lo envía a la SET — todo en una sola llamada.
POST /api/v1/documentos/erp 
{
  // ── TIPO DE DOCUMENTO ──────────────────────────────────────────────────────
  "tipoDocumento": 1,            // 1=Factura, 4=Autofactura, 5=Nota crédito, 6=Nota débito, 7=Remisión

  // ── FECHA Y MONEDA ─────────────────────────────────────────────────────────
  "fecha": "2025-04-03T10:30:00", // ISO 8601. Si no se envía, se usa la hora actual del servidor
  "moneda": "PYG",               // PYG (defecto) | USD | BRL | EUR | ARS
  "condicionVenta": 1,           // 1=Contado (defecto) | 2=Crédito

  // ── CLIENTE / RECEPTOR ─────────────────────────────────────────────────────
  "cliente": {
    "tipo": 1,                   // 1=RUC | 2=CI paraguaya | 3=Pasaporte/extranjero | 4=Sin doc (innominado)
    "documento": "80012345-6",   // RUC con dígito verificador, o CI. No enviar si tipo=4
    "razonSocial": "Empresa SA", // Nombre completo o razón social. No enviar si tipo=4
    "email": "fe@empresa.py",    // Opcional — NODO envía el KUDE por email a esta dirección
    "telefono": "021555000",     // Opcional
    "direccion": "Av. 1234"      // Opcional
  },

  // ── ÍTEMS / LÍNEAS DE DETALLE ──────────────────────────────────────────────
  "items": [
    {
      "descripcion": "Notebook HP 15",  // Descripción del bien o servicio
      "cantidad": 2,                    // Puede ser decimal: 1.5
      "precioUnitario": 2500000,        // Sin IVA incluido
      "precioTotal": 5000000,           // cantidad × precioUnitario (el motor valida coherencia)
      "tasaIVA": 10,                    // 0=Exento | 5=IVA 5% | 10=IVA 10%
      "unidadMedida": 77,               // Código SET: 77=Unidad, 83=Servicio. Ver tabla completa abajo
      "codigoProducto": "NB-HP-15"      // Opcional — código interno del ERP
    },
    {
      "descripcion": "Garantía extendida",
      "cantidad": 1,
      "precioUnitario": 200000,
      "precioTotal": 200000,
      "tasaIVA": 5                      // Mismo comprobante puede tener ítems con distintas tasas
    }
  ],

  // ── TRAZABILIDAD Y NOTIFICACIONES ──────────────────────────────────────────
  "referenciaExterna": "FAC-2025-0042", // Tu ID interno — NODO lo devuelve en el webhook para que sepas qué factura es
  "webhookUrl": "https://mi-erp.py/webhook/nodo" // Opcional — NODO notifica a esta URL cuando la SET responde
}
✅ Respuesta — Aprobado (HTTP 201)
{
  "ok": true,
  "data": {
    "cdc": "01803778760...", // 44 dígitos — GUARDAR
    "numero": "001-002-0000001",
    "estado": "aprobado",
    "sifen_codigo": "0260",
    "sifen_mensaje": "Documento aprobado"
  }
}
❌ Respuesta — Rechazado (HTTP 201)
{
  "ok": true,
  "data": {
    "cdc": "01803778760...",
    "numero": "001-002-0000002",
    "estado": "rechazado",
    "sifen_codigo": "0300",    // Ver tabla errores SET
    "sifen_mensaje": "RUC inválido"
  }
}
Importante sobre documentos rechazados: cuando la SET rechaza un documento, igual se retorna HTTP 201 con estado: "rechazado". El documento queda registrado en el sistema con el número asignado, pero no tiene validez fiscal. Si necesitás corregirlo y reemitirlo, debés crear uno nuevo (con nuevo número secuencial) — no se puede reenviar el mismo.
Emitir una nota de crédito
La nota de crédito anula o ajusta una factura previamente emitida. Usa el mismo endpoint que la factura, pero con tipoDocumento: 5 y referenciando la factura original con su CDC.
POST /api/v1/documentos/erp 
{
  "tipoDocumento": 5,            // 5 = Nota de crédito electrónica
  "fecha": "2025-04-05T09:00:00",
  "moneda": "PYG",

  // ── DOCUMENTO QUE SE ESTÁ AJUSTANDO ───────────────────────────────────────
  "documentoAsociado": {
    "cdc": "01803778760001002000001...", // CDC de la factura original que se está anulando/ajustando
    "tipoDocumentoAsociado": 1          // 1 = Factura (tipo del doc original)
  },

  // ── MOTIVO DE EMISIÓN ──────────────────────────────────────────────────────
  "motivoEmision": 1,            // 1=Devolución | 2=Anulación | 3=Descuento | 4=Bonificación
                                 // 5=Crédito incobrable | 6=Recupero costo | 7=Recupero gasto | 8=Ajuste precio

  // ── MISMO RECEPTOR QUE LA FACTURA ORIGINAL ────────────────────────────────
  "cliente": {
    "tipo": 1,
    "documento": "80012345-6",
    "razonSocial": "Empresa SA"
  },

  // ── LOS ÍTEMS QUE SE DEVUELVEN O AJUSTAN ──────────────────────────────────
  "items": [
    {
      "descripcion": "Devolución Notebook HP 15",
      "cantidad": 1,              // Cantidad que se devuelve (no tiene que ser el total)
      "precioUnitario": 2500000,
      "precioTotal": 2500000,
      "tasaIVA": 10
    }
  ],

  "referenciaExterna": "NC-2025-0001"
}
⚠️ La nota de crédito no cancela automáticamente la factura original en la SET. Simplemente crea un nuevo documento que ajusta el crédito fiscal. Para anular completamente una factura aprobada, usá el endpoint de cancelación (ver tab Consultar).
Consultar y gestionar documentos
Consultar un documento por CDC
Devuelve el estado actual y todos los datos del documento. Útil para verificar si fue aprobado después de un timeout.
GET /api/v1/documentos/{cdc} 
// {cdc} = los 44 dígitos del CDC del documento
// No lleva body — solo el header X-API-Key

// Respuesta:
{
  "ok": true,
  "data": {
    "cdc":           "01803778760001002000001...",
    "numero":        "001-002-0000001",
    "tipo":          1,
    "estado":        "aprobado",           // aprobado | rechazado | pendiente | cancelado
    "sifen_codigo":  "0260",
    "sifen_mensaje": "Documento aprobado",
    "monto_total":   5200000,
    "monto_iva_10":  472727,
    "monto_iva_5":   9524,
    "receptor_razon":"Empresa SA",
    "receptor_doc":  "80012345-6",
    "creado_en":     "2025-04-03T10:30:45Z"
  }
}
Buscar por referencia externa (tu ID interno)
Si no guardaste el CDC pero sí usaste referenciaExterna al emitir, podés buscarlo así.
GET /api/v1/documentos?referenciaExterna=FAC-2025-0042 
// También podés combinar filtros:
GET /api/v1/documentos?estado=rechazado&desde=2025-04-01&hasta=2025-04-30
GET /api/v1/documentos?tipo=5           // Solo notas de crédito
GET /api/v1/documentos?limit=50&offset=0  // Paginación
Cancelar un documento aprobado
Solo se pueden cancelar documentos con estado aprobado. La cancelación se registra en la SET.
POST /api/v1/documentos/{cdc}/cancelar 
{
  "motivo": "Error en datos del cliente"  // Motivo obligatorio para registrar en el log
}

// Respuesta exitosa:
{ "ok": true, "data": { "cdc": "...", "estado": "cancelado" } }
Descargar KUDE (PDF) y XML firmado
El KUDE es el comprobante oficial en PDF. El XML firmado es el archivo que queda registrado en la SET. Ambos solo están disponibles para documentos con estado: "aprobado".
Descargar KUDE en PDF
GET /api/v1/documentos/{cdc}/kude?formato=a4 
// Parámetros:
// formato = a4 (defecto) | carta | ticket

// Respuesta: binario PDF (Content-Type: application/pdf)
// Importante: este endpoint requiere el header X-API-Key igual que el resto
// No es una URL pública — no funciona abriéndola directo en el browser sin la key

// Ejemplo en código (ver tab Ejemplos para más lenguajes):
response = GET /api/v1/documentos/01803778760.../kude?formato=a4
           X-API-Key: sk_live_XXXX
// → guardar bytes como archivo .pdf o mostrar en un iframe con blob URL
Descargar XML firmado
GET /api/v1/documentos/{cdc}/xml 
// Devuelve el XML firmado (Content-Type: application/xml)
// Este es el archivo que se presentó a la SET y quedó aprobado
// Útil para archivado fiscal o para sistemas que requieren el XML original
Envío automático por email: si el objeto cliente incluye el campo email, NODO envía automáticamente el KUDE en PDF al receptor al momento de ser aprobado (si el emisor tiene SMTP configurado). No es necesario que el ERP descargue y reenvíe el PDF manualmente.
Webhooks — notificaciones en tiempo real
Si enviás el campo webhookUrl al emitir el documento, NODO hace un POST a esa URL cuando la SET responde. Es opcional pero muy recomendado — así tu sistema se actualiza sin tener que hacer polling.
Lo que recibe tu endpoint
POST → https://mi-erp.py/webhook/nodo  ·  Content-Type: application/json 
{
  "evento":    "de.aprobado",             // de.aprobado | de.rechazado | de.cancelado
  "timestamp": "2025-04-03T10:30:45Z",
  "documento": {
    "id":                "uuid-interno",
    "cdc":               "01803778760001002000001...",
    "numero":            "001-002-0000001",
    "tipoDocumento":     1,
    "estado":            "aprobado",
    "referenciaExterna": "FAC-2025-0042",  // ← tu ID interno — con esto actualizás tu BD
    "receptor": {
      "documento":   "80012345-6",
      "razonSocial": "Empresa SA"
    },
    "montoTotal":  5200000,
    "montoIVA10":  472727,
    "montoIVA5":   9524,
    "sifen": {
      "codigo":       "0260",
      "mensaje":      "Documento aprobado",
      "enviadoEn":    "2025-04-03T10:30:30Z",
      "respondidoEn": "2025-04-03T10:30:45Z"
    }
  }
}
¿Qué debe hacer tu endpoint al recibir el webhook?
// 1. Responder HTTP 200 inmediatamente (antes de procesar)
// Si no respondés en 10 segundos, NODO lo intenta de nuevo

// 2. Procesar en background:
// - Actualizar el estado de la factura en tu BD usando referenciaExterna
// - Guardar el CDC si aún no lo tenías
// - Si fue rechazado, alertar al usuario o reintentar con datos corregidos

// Ejemplo mínimo (Node.js / Express):
app.post('/webhook/nodo', (req, res) => {
  res.sendStatus(200)  // Responder primero
  const { evento, documento } = req.body
  if (evento === 'de.aprobado') {
    db.update('facturas', {
      where: { referencia_interna: documento.referenciaExterna },
      set:   { cdc: documento.cdc, estado: 'aprobada' }
    })
  }
})
Reintentos automáticos
Si tu endpoint no responde o devuelve error, NODO reintenta con backoff: 10s → 30s → 2min → 10min → 1h. Máximo 5 intentos.
Idempotencia
Tu endpoint puede recibir el mismo webhook más de una vez por los reintentos. Usá el cdc como clave única para evitar procesar duplicados.
Manejo de errores
Errores HTTP de la API NODO
Código HTTPSignificadoQué hacer
400 Bad Request El body del request tiene errores de formato (JSON inválido, campo faltante obligatorio) Revisar la estructura del JSON. El mensaje de error indica el campo exacto.
401 Unauthorized El header X-API-Key no está presente o la clave es inválida Verificar que se envía el header en cada request. Verificar que la key está activa en el panel.
403 Forbidden La API Key fue revocada o no tiene permisos para este tenant Generar una nueva API Key desde el panel → Integración ERP.
422 Unprocessable Los datos son válidos en formato pero inválidos en lógica (RUC mal formado, monto negativo, tasaIVA no permitida) Revisar los valores de los campos. El mensaje indica qué falló.
429 Too Many Requests Se superó el límite de 100 requests por minuto Implementar queue o throttling en el ERP. Reintentar después de 60 segundos.
500 Internal Error Error interno del motor. Puede ser transitorio. Reintentar después de 30 segundos. Si persiste, contactar soporte con el timestamp del error.

Código SETSignificadoSolución
0260✅ AprobadoTodo ok. Entregar el KUDE al cliente.
0261⚠️ Aprobado con observacionesEl documento es válido pero tiene advertencias. Revisar sifen_mensaje.
0300Datos incorrectosAlgún campo no pasa la validación de la SET. El mensaje indica cuál. Corregir y reemitir con nuevo número.
0302CSC inválidoEl Código de Seguridad del Contribuyente no coincide con el de la SET. Verificar en el panel → Configuración.
0303Timbrado inactivo o vencidoRenovar el timbrado en Marangatú y actualizar en el panel → Configuración.
0422CDC duplicadoEl documento ya fue procesado. No reintentar — consultar el estado con el CDC existente.
0500Error interno SETProblema del lado de la SET. Reintentar en 5-10 minutos.

Cálculo del IVA: el motor calcula el IVA automáticamente del precioTotal.
IVA 10%: monto_iva = precioTotal / 11  ·  IVA 5%: monto_iva = precioTotal / 21
No es necesario enviar el monto del IVA — se calcula y se verifica internamente.
Ejemplos de código por lenguaje
Copiar y adaptar según el lenguaje del ERP.
JavaScript / Node.js
// Emitir factura
const NODO_URL = process.env.NODO_API_URL
const NODO_KEY = process.env.NODO_API_KEY

async function emitirFactura(datos) {
  const res = await fetch(`${NODO_URL}/documentos/erp`, {
    method:  'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-API-Key':    NODO_KEY,
    },
    body: JSON.stringify({
      tipoDocumento:     1,
      cliente:           datos.cliente,
      items:             datos.items,
      referenciaExterna: datos.numeroInterno,
      webhookUrl:        'https://mi-erp.py/webhook/nodo',
    }),
  })

  const result = await res.json()

  if (!res.ok) throw new Error(result.error)

  // Guardar en BD independientemente del estado
  await db.facturas.update({
    where: { id: datos.numeroInterno },
    data:  { cdc: result.data.cdc, estado_set: result.data.estado },
  })

  return result.data
}

async function descargarKude(cdc) {
  const res = await fetch(`${NODO_URL}/documentos/${cdc}/kude?formato=a4`, {
    headers: { 'X-API-Key': NODO_KEY },
  })
  if (!res.ok) throw new Error('Error descargando KUDE')

}

function emitirFacturaNodo($cliente, $items, $referencia) {
    $payload = json_encode([
        'tipoDocumento'     => 1,
        'cliente'           => $cliente,
        'items'             => $items,
        'referenciaExterna' => $referencia,
    ]);

    $ch = curl_init($_ENV['NODO_API_URL'] . '/documentos/erp');
    curl_setopt_array($ch, [
        CURLOPT_POST           => true,
        CURLOPT_POSTFIELDS     => $payload,
        CURLOPT_RETURNTRANSFER => true,
        CURLOPT_HTTPHEADER     => [
            'Content-Type: application/json',
            'X-API-Key: ' . $_ENV['NODO_API_KEY'],
        ],
    ]);

    $respuesta = json_decode(curl_exec($ch), true);
    $httpCode  = curl_getinfo($ch, CURLINFO_HTTP_CODE);
    curl_close($ch);

    if ($httpCode >= 400) {
        throw new Exception($respuesta['error'] ?? 'Error de API');
    }

    return $respuesta['data'];
}
