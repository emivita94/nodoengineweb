🗄 Conexión con Supabase — paso a paso

⚠️
Si el botón de login no hace nada o ves datos de ejemplo, es porque js/config.js todavía tiene las credenciales de plantilla. Seguí estos pasos.

Paso 1 — Obtener las credenciales de Supabase

1
Entrá a tu proyecto en Supabase
Ir a supabase.com/dashboard → elegí tu proyecto → en el menú izquierdo click en Settings → API.
2
Copiar Project URL y Anon Key
Vas a ver dos valores importantes:
• Project URL — ejemplo: https://abcdefgh.supabase.co
• anon / public key — empieza con eyJhbGci... (es larga, copiá todo)
3
Editar js/config.js
Abrí el archivo con cualquier editor de texto (Notepad, VS Code, etc.) y reemplazá los valores:
export const CONFIG = {
  // Pegá tu Project URL aquí ↓
  SUPABASE_URL:  'https://abcdefghijklmnop.supabase.co',

  // Pegá tu anon key aquí ↓ (la clave larga)
  SUPABASE_ANON: 'eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc...',

  // URL de tu API en Railway (para firmar y enviar a SET)
  API_URL: 'https://tu-nodo-engine.railway.app/api/v1',
};

Paso 3 — Crear bucket de logos en Supabase Storage

1
Storage → New bucket
En el menú izquierdo de Supabase → Storage → botón New bucket
2
Configurar el bucket
• Name: logos
• Public bucket: ✅ activar
• Click Save
3
Política de acceso (RLS)
En Storage → Policies → crear una política que permita INSERT y SELECT para usuarios autenticados. O en SQL Editor:
-- Permitir subir logos (usuarios autenticados)
CREATE POLICY "logos_upload" ON storage.objects
  FOR INSERT WITH CHECK (
    bucket_id = 'logos' AND auth.role() = 'authenticated'
  );

-- Permitir ver logos (público)
CREATE POLICY "logos_public" ON storage.objects
  FOR SELECT USING (bucket_id = 'logos');

-- Permitir reemplazar logos
CREATE POLICY "logos_update" ON storage.objects
  FOR UPDATE USING (
    bucket_id = 'logos' AND auth.role() = 'authenticated'
  );

Paso 4 — Configurar RLS en la tabla tenants

Para que el panel pueda leer y escribir tenants, ejecutá en SQL Editor:

-- Habilitar RLS
ALTER TABLE tenants ENABLE ROW LEVEL SECURITY;

-- Solo usuarios autenticados pueden ver tenants
CREATE POLICY "tenants_select" ON tenants
  FOR SELECT USING (auth.role() = 'authenticated');

-- Solo usuarios autenticados pueden crear tenants
CREATE POLICY "tenants_insert" ON tenants
  FOR INSERT WITH CHECK (auth.role() = 'authenticated');

-- Solo usuarios autenticados pueden actualizar tenants
CREATE POLICY "tenants_update" ON tenants
  FOR UPDATE USING (auth.role() = 'authenticated');

Paso 5 — Crear usuario admin en Supabase Auth

En Supabase Dashboard → Authentication → Users → Add user → Create new user:

  • Email: info@nodoinformatica.com
  • Password: Abc1234*
  • Activar Auto Confirm User ✅
✅
Con esto hecho, el login con info@nodoinformatica.com / Abc1234* va a autenticar contra Supabase real. Los tenants que crees van a guardarse en la base de datos.

Paso 6 — Subir config.js a Cloudflare (producción)

El archivo config.js no va al repo por seguridad. Para producción en Cloudflare Pages tenés dos opciones:

Opción A — Build step con Vite (recomendado para producción):

# En Cloudflare Pages → Settings → Environment Variables:
VITE_SUPABASE_URL  = https://tuproyecto.supabase.co
VITE_SUPABASE_ANON = eyJhbGci...
VITE_API_URL       = https://tu-api.railway.app/api/v1

Opción B — Config hardcodeada para staging:
Incluir config.js con los valores reales solo en el entorno de staging. En producción usar variables de entorno.

🔧 Manual de errores y soluciones

Los errores más frecuentes del sistema, organizados por categoría.

🔑 Autenticación y configuración

🔴
El botón de login no hace nada
Configuración ▼

Causa: js/config.js tiene los valores de plantilla (TUPROYECTO.supabase.co), el módulo falla al importar y el script se cae silenciosamente.

Abrí js/config.js y reemplazá TUPROYECTO con tu Project URL real de Supabase.
Verificá que el Anon Key no tenga TUANON_KEY_AQUI — copiá la clave completa desde Supabase → Settings → API.
Mientras tanto podés entrar con info@nodoinformatica.com / Abc1234* — esas credenciales funcionan sin Supabase.
🔴
Error: "Invalid login credentials"
Auth ▼

Causa: Email o contraseña incorrectos en Supabase Auth, o el usuario no existe.

Verificar que el usuario exista en Supabase → Authentication → Users.
Si el usuario existe, usar "Reset password" desde el dashboard de Supabase.
Verificar que el email esté confirmado (columna "Confirmed At" no vacía).
🟡
El panel muestra datos de ejemplo en vez de los reales
Configuración ▼

Causa: El sistema detecta que config.js no tiene credenciales válidas y activa el modo demo automáticamente.

Completar js/config.js con la URL y Anon Key reales de Supabase.
Abrir la consola del navegador (F12) y buscar mensajes que digan [NODO] Modo demo.
Verificar que la tabla tenants exista en Supabase y tenga al menos una fila.

📡 Conexión API y Supabase

🔴
Error: "Failed to fetch" al cargar empresas
API ▼

Causa: No hay conexión con Supabase. Puede ser URL incorrecta, CORS, o problema de red.

Verificar que SUPABASE_URL en config.js no tenga barra al final: correcto https://abc.supabase.co, incorrecto https://abc.supabase.co/.
En Supabase → Settings → API → verificar que el proyecto esté activo (no pausado por inactividad).
Abrir F12 → Network → buscar la llamada a supabase.co → ver el error exacto.
🔴
Error 403: "Row Level Security policy violated"
RLS ▼

Causa: Las políticas de seguridad de Supabase (RLS) están bloqueando el acceso a la tabla.

Ejecutar el SQL de políticas RLS del Paso 4 de la guía de conexión.
Verificar que el token de autenticación sea válido — cerrar sesión y volver a entrar.
Temporalmente para testing, desactivar RLS: ALTER TABLE tenants DISABLE ROW LEVEL SECURITY; — volver a activar antes de producción.
🟡
Error al subir logo: "Bucket not found"
Storage ▼

Causa: El bucket logos no fue creado en Supabase Storage.

Ir a Supabase → Storage → New bucket → nombre logos → activar Public → Save.
Verificar que el nombre sea exactamente logos (minúsculas).

🏛 Errores SET / SIFEN

🔴
Código SET 0300 — "Documento con datos incorrectos"
SIFEN ▼

Causa: El XML enviado tiene datos que no coinciden con lo registrado en SET (RUC, monto IVA, timbrado, etc.).

Verificar que el RUC del receptor sea válido y tenga el formato correcto XXXXXXXX-D.
Verificar que el cálculo del IVA sea correcto: IVA 10% = monto / 11, IVA 5% = monto / 21.
Verificar que el timbrado esté vigente (fecha actual entre vigencia_desde y vigencia_hasta).
Revisar el XML rechazado en el log del lote — SET indica el campo exacto con el error.
🔴
Código SET 0302 — "CSC inválido"
SIFEN ▼

Causa: El Código de Seguridad del Contribuyente (CSC) configurado no coincide con el registrado en la SET.

Verificar el CSC y ID CSC en la configuración de la empresa — ir al panel → Configuración → Seguridad SIFEN.
El CSC se obtiene en Marangatú → Servicios Electrónicos → Código de Seguridad.
Verificar que el ambiente (Test/Producción) coincida con el CSC — cada ambiente tiene su propio CSC.
🔴
Código SET 0303 — "Timbrado no encontrado o inactivo"
SIFEN ▼

Causa: El número de timbrado no existe, venció, o fue dado de baja en SET.

Verificar que el timbrado esté activo en Marangatú.
Verificar las fechas de vigencia en panel → Configuración → Puntos de expedición.
Si el timbrado venció, tramitar uno nuevo en Marangatú y cargarlo en el panel.
🟡
Código SET 0422 — "CDC ya registrado"
SIFEN ▼

Causa: Se intentó enviar un documento con un CDC que ya existe en la SET (documento duplicado).

No reintentar — el documento ya fue procesado. Consultar su estado con el CDC en el panel.
Verificar que el número secuencial no esté repetido en la tabla timbrados.

🔑 Certificado digital

🔴
Error al firmar: "Certificado vencido"
Certificado ▼

Causa: El certificado digital PKCS#12 (.p12) expiró.

Tramitar un nuevo certificado ante la Cámara de la Industria y Comercio o la SET.
Una vez obtenido, subirlo en panel → Configuración → Certificado digital.
Activar las alertas de vencimiento en Configuración de correo para recibir aviso 30 días antes.
🔴
Error al firmar: "Contraseña del certificado incorrecta"
Certificado ▼

Causa: La contraseña guardada en el sistema no coincide con la del archivo .p12.

Ir a panel → Configuración → Certificado digital → actualizar la contraseña.
Verificar la contraseña abriendo el .p12 localmente con el Administrador de certificados de Windows.

🌐 Cloudflare / Deploy

🔴
"Could not find wrangler.json" al hacer deploy
Cloudflare ▼

Causa: El proyecto fue creado como Worker en vez de Pages.

Borrar el proyecto en Cloudflare → Workers & Pages → Settings → Delete project.
Crear uno nuevo: Workers & Pages → Create application → tab Pages (no Workers) → Connect to Git.
En la configuración de build: Framework = None, Build command = vacío, Output = /.
🟡
El deploy funciona pero los cambios no aparecen
Cloudflare ▼

Causa: Cache del navegador o de Cloudflare.

Hacer Ctrl+Shift+R (hard refresh) en el navegador.
En Cloudflare → tu proyecto → Deployments → verificar que el último deploy haya terminado con ✅.
Si el problema persiste, en Cloudflare → Caching → Purge Everything.

🔌 Integración con sistemas de terceros

Guía paso a paso para conectar cualquier ERP, sistema de punto de venta o aplicación con el motor NODO Engine.

✅
La integración funciona con cualquier sistema que pueda hacer llamadas HTTP: SAP, sistemas propios, Excel con macros, Python, PHP, Node.js, etc.

Flujo general de integración

Tu sistema genera una venta → POST a /api/v1/documentos/erp → NODO firma y envía a SET → SET aprueba/rechaza → NODO notifica a tu sistema por webhook → Tu sistema muestra el resultado

Paso 1 — Registrar la empresa en el panel

1
Crear el tenant en el panel NODO
Ir a panel.nodoengine.py → click + Nueva empresa → completar RUC, razón social y ambiente.
2
Cargar el certificado digital
Dentro de la empresa → Configuración → Certificado digital → subir el .p12 con alias y contraseña.
3
Configurar el timbrado
Ir a Configuración → Puntos de expedición → agregar establecimiento (001), punto (001), número de timbrado y vigencia.
4
Obtener la API Key
Ir a API Keys → + Nueva API Key → elegir ambiente → copiar la key generada (aparece solo una vez).

Paso 2 — Configurar en tu sistema

En tu ERP o sistema, guardar estos dos datos en la configuración de facturación electrónica:

ParámetroValor
URL del serviciohttps://tu-api.railway.app/api/v1
API Keysk_live_XXXXXXXXXXXXXXXX
Endpoint facturasPOST /documentos/erp
Header de authX-API-Key: sk_live_XXXX

Paso 3 — Emitir una factura desde tu sistema

Cuando el cliente confirma una compra, tu sistema hace esta llamada:

POST https://tu-api.railway.app/api/v1/documentos/erp
X-API-Key: sk_live_XXXXXXXXXXXXXXXX
Content-Type: application/json

{
  "tipoDocumento": 1,
  "cliente": {
    "tipo": 1,
    "documento": "4567890",
    "razonSocial": "Juan Pérez"
  },
  "items": [
    {
      "descripcion":    "Producto A",
      "cantidad":        1,
      "precioUnitario":  150000,
      "precioTotal":     150000,
      "tasaIVA":         10
    }
  ],
  "referenciaExterna": "VENTA-2025-001",
  "webhookUrl": "https://tu-sistema.com/webhook/fe"
}

Paso 4 — Interpretar la respuesta

NODO responde inmediatamente con el estado del documento:

// Respuesta exitosa (201 Created)
{
  "ok": true,
  "data": {
    "cdc":     "01800123456001001000012300920251234567890",
    "numero": "001-001-0000001",
    "estado": "aprobado",
    "sifen_codigo": "0260",
    "sifen_mensaje": "Documento aprobado"
  }
}

// Respuesta con error de datos (422)
{
  "ok": false,
  "error": "RUC del receptor inválido"
}

Paso 5 — Recibir el webhook

Si pasaste webhookUrl, NODO notifica a tu sistema cuando el estado cambia:

// NODO envía POST a tu-sistema.com/webhook/fe
{
  "evento":  "de.aprobado",
  "cdc":     "01800123456001001000012300920251234567890",
  "numero": "001-001-0000001",
  "estado": "aprobado",
  "referenciaExterna": "VENTA-2025-001",
  "timestamp": "2025-04-04T10:05:23Z"
}

// Tu sistema responde con HTTP 200 para confirmar recepción

Paso 6 — Descargar el KUDE (PDF de la factura)

Para obtener el PDF de la factura aprobada:

GET /documentos/{cdc}/kude?formato=a4
X-API-Key: sk_live_XXXX

// Devuelve el PDF en binario (Content-Type: application/pdf)
// Podés descargarlo o enviarlo por email al cliente

Referencia de campos del request

CampoTipoObligatorioDescripción
tipoDocumentonumber✅1=Factura, 5=Nota crédito, 6=Nota débito, 7=Remisión
cliente.tiponumber✅1=RUC, 2=CI, 3=Pasaporte, 4=Innominado
cliente.documentostringSi tipo ≠ 4Número de RUC o CI (sin puntos ni guiones para CI)
cliente.razonSocialstringSi tipo ≠ 4Nombre o razón social del receptor
itemsarray✅Mínimo 1 item
items[].tasaIVAnumber✅0, 5 o 10
referenciaExternastring❌ID de tu sistema (para buscar después)
webhookUrlstring❌URL donde NODO notifica el resultado
fechastring❌ISO 8601 — por defecto es la fecha actual

Códigos de respuesta SET

CódigoSignificadoAcción
0260✅ AprobadoTodo ok — entregar al cliente
0261⚠️ Aprobado con obs.Aprobado pero revisar observaciones
0300❌ Datos incorrectosRevisar campos del documento
0302❌ CSC inválidoVerificar CSC en configuración
0303❌ Timbrado inactivoRenovar timbrado en Marangatú
0422⚠️ CDC duplicadoYa fue procesado — no reintentar
0500❌ Error SETProblema del lado de SET — reintentar en unos minutos

Ejemplos de integración por lenguaje

Python:

import requests

response = requests.post(
    'https://tu-api.railway.app/api/v1/documentos/erp',
    headers={'X-API-Key': 'sk_live_XXXX'},
    json={
        'tipoDocumento': 1,
        'cliente': {'tipo': 2, 'documento': '4567890', 'razonSocial': 'Juan Pérez'},
        'items': [{'descripcion': 'Producto', 'cantidad': 1,
                    'precioUnitario': 150000, 'precioTotal': 150000, 'tasaIVA': 10}],
        'referenciaExterna': 'VENTA-001'
    }
)
data = response.json()
print(data['data']['cdc'])  # CDC del documento aprobado
